# Sippomi iOS Swipe App Plan

## 1. 現状の前提

- 現在の `Sippomi` は Vite ベースの静的サイト。
- 条件入力は `species / gender / vibe / color` のフィルタ式。
- 推薦ロジックは [`src/diagnosis.js`](/Users/shimizutaiga/Projects/apps/sippomi/src/diagnosis.js) にあり、条件一致とスコア計算で一覧を返す。
- データは [`data/names.json`](/Users/shimizutaiga/Projects/apps/sippomi/data/names.json) にあり、約 1,200 件の名前候補を持つ。
- Supabase を使ったお気に入り保存の土台がある。

このため、iOS 版は「推薦ロジックを捨てて作り直す」よりも、「初期フィルタは維持しつつ、出力 UI を一覧からスワイプ式に置き換える」方が速くて安全。

## 2. 目指す体験

一覧を返す診断アプリではなく、次の流れにする。

1. Step 1: ペットの種類を選ぶ
2. Step 2: 性別、毛色、必要なら体格や文字数を選ぶ
3. Step 3: かわいい、和風、上品などの雰囲気を選ぶ
4. Step 4: 名前の方向性を選ぶ
5. 初回候補を 20-40 件生成する
6. Tinder ライクなカードを左右スワイプして `Like / Pass`
7. スワイプ履歴から好みベクトルを更新する
8. 「あなたは和風・短め・メス寄りが好き」などの好みサマリーを表示する
9. 最後にベスト候補、お気に入り、比較結果を表示する

## 3. 推奨アーキテクチャ

### 方針

MVP は `Expo + React Native + TypeScript + Supabase` を推奨。

理由:

- 既存が JavaScript なので推薦ロジックを移植しやすい
- iOS を先に出しつつ Android 展開も後でしやすい
- スワイプ UI は既存ライブラリか Gesture Handler で十分実装可能
- App Store 審査用のネイティブ感も十分確保できる

### 構成

- `apps/web`: 現行 Vite サイト
- `apps/ios`: Expo iOS アプリ
- `packages/recommendation-core`: 既存の `constants.js` と `diagnosis.js` を再編した共通推薦ロジック
- `Supabase`: 認証、お気に入り、スワイプ履歴、セッション保存

### なぜ SwiftUI 単独ではなく Expo か

SwiftUI のほうが iOS らしさは強いが、現状の資産活用と速度を考えると MVP は Expo が優位。  
ただし 6 か月以内に本格的に iOS ネイティブ体験へ寄せるなら、将来の v2 で SwiftUI 化する余地は残す。

## 4. 画面設計

### A. Onboarding

- 価値訴求
- 「3 分で名前候補が絞れる」
- ログインなしで開始

### B. Step 1-4 インテーク

- Step 1: 種類
- Step 2: 性別、毛色
- Step 3: 雰囲気タグ
- Step 4: 文字数、和風/洋風、個性的さ

ポイント:

- 初回で聞きすぎない
- 必須は `種類` のみ
- 他はスキップ可にして離脱を防ぐ

### C. Swipe Deck

- 中央に 1 枚カード
- 右スワイプ: Like
- 左スワイプ: Pass
- 下スワイプ or ボタン: 保留
- タップ: 詳細表示

カード表示要素:

- 名前
- 読み
- 意味
- 雰囲気タグ
- おすすめ理由

### D. Preference Insight

スワイプ 5 件ごとに更新。

- 好みの雰囲気 Top 3
- 好みの文字数
- 好みの色傾向
- 好みの性別傾向
- 「最近の傾向」 と 「全体傾向」

例:

- あなたは `和風`
- `2-3文字`
- `かわいい`
- `白系イメージ`
の名前を好む傾向です

### E. Result / Save

- 最終候補 Top 10
- Like 一覧
- 2件比較
- お気に入り保存
- シェア
- 「別の方向でもう一度探す」

## 5. 推薦ロジック

### 初回候補生成

初回は既存ロジックを使う。

- 種類は hard filter
- 性別、毛色、雰囲気は soft filter を含む
- 完全一致だけでなく近い候補も混ぜる

### スワイプ後の再ランキング

各名前にある属性を重み化する。

- `species`
- `gender`
- `color`
- `vibe`
- `length`
- `theme`

Like された属性の重みを上げ、Pass された属性の重みを下げる。

### 重要な設計

- `exploration` を 20-30% 残す
- 同じ系統ばかり出さない `diversity 制約` を入れる
- 直近 5 件の反応をやや強めに反映する
- 同じ名前は再表示しない

### スコア例

`totalScore = initialFit * 0.5 + learnedPreference * 0.4 + diversityBoost * 0.1`

MVP ではこの程度で十分。  
いきなり機械学習モデルを入れる必要はない。

## 6. データ設計

追加推奨テーブル:

- `naming_sessions`
- `swipe_events`
- `preference_profiles`
- `favorite_names`

### naming_sessions

- `id`
- `user_id nullable`
- `species`
- `initial_answers jsonb`
- `started_at`
- `completed_at`

### swipe_events

- `id`
- `session_id`
- `name_id`
- `action` (`like`, `pass`, `hold`)
- `shown_rank`
- `created_at`

### preference_profiles

- `id`
- `user_id or session_id`
- `preference_vector jsonb`
- `summary jsonb`
- `updated_at`

### 保存方針

- 未ログインでも匿名 session で開始
- 保存や履歴同期の時だけログイン誘導
- これで App Review 的にも UX 的にも通しやすい

## 7. App Review 観点

## 結論

この案は、設計と表現を丁寧にすれば App Store 審査は十分通せる可能性が高い。  
ただし「スワイプ UI」自体は問題ではなく、以下を外すと落ちやすい。

### 通しやすい理由

- ペット名提案は安全性が高い
- 出会い系や成人向けではない
- デジタルコンテンツ課金も標準的に設計できる
- 既存の推薦ロジックと保存機能があり、単なる LP ラッパーでは終わらせやすい

### 落ちやすいポイント

1. Web サイトの薄いラッパーに見える
2. AI や診断の表現が誇大
3. ログイン必須にしているのに、ログインなしでも成立する内容
4. 課金導線が外部決済前提
5. プライバシー説明が雑

### 主要ガイドラインとの対応

- `2.1 App Completeness`
  - 審査時に動くバックエンド、デモアカウント、テスト可能な導線が必要
- `2.3.1 Accurate Metadata`
  - スクリーンショットや説明文で「できること」を盛らない
- `3.1.1 In-App Purchase`
  - プレミアム候補、追加デッキ、無制限保存などを売るなら Apple の IAP を使う
- `4.2 Minimum Functionality`
  - 単なるサイト埋め込みではなく、スワイプ学習、保存、履歴、比較などアプリらしい価値を入れる
- `5.1.1 Data Collection and Storage`
  - 何を収集するか、削除方法、退会方法を明示する
- `1.1.6 False information and features`
  - 「AIが最適名を保証」など、保証的な言い方は避ける

### 審査を通しやすくする具体策

- ログインなしで開始可能にする
- スワイプで好み学習されることを初回に説明する
- 課金なしでも十分遊べる導線を用意する
- 審査用ノートに
  - 基本フロー
  - どこで課金が出るか
  - ログインが必要な場面
  - デモ用アカウント
  を明記する

## 8. 収益モデル

### おすすめ順

1. フリーミアム
2. 週額または月額サブスク
3. 単発課金

### MVP に合う商品

- 無料:
  - 初回診断
  - 1日 20 スワイプ
  - お気に入り 10 件まで
- 有料買い切り:
  - 無制限スワイプ
  - 詳細な好み分析
  - 家族共有用の比較ボード
- 有料サブスク:
  - 新着名前テーマ
  - 名前辞典の追加
  - 保存数無制限
  - AI コメント強化

### 価格の現実案

- 買い切り: `¥600 - ¥1,200`
- 月額: `¥300 - ¥680`
- 年額: `¥3,000 - ¥5,000`

最初は買い切り 1 本のほうが、価値が伝わりやすく解約ストレスも少ない。  
名前決めは短期ユースなので、いきなり月額一本はやや弱い。

## 9. 収益試算

以下は推定であり、実績ではない。

### 前提

- 月間新規 DL
  - 小規模: 3,000
  - 中規模: 10,000
  - 好調: 30,000
- 有料転換率
  - 弱い: 1.0%
  - 標準: 2.0%
  - 良い: 3.5%
- ARPPU
  - 買い切り中心: ¥900
  - サブスク中心: 初月実現売上 ¥650
- Apple 手数料は小規模事業者プログラム適用時 15% で試算

### 買い切りモデルの月商イメージ

- 3,000 DL × 2.0% × ¥900 = `¥54,000` 粗売上
- 手数料後 = 約 `¥45,900`

- 10,000 DL × 2.0% × ¥900 = `¥180,000` 粗売上
- 手数料後 = 約 `¥153,000`

- 30,000 DL × 3.5% × ¥900 = `¥945,000` 粗売上
- 手数料後 = 約 `¥803,250`

### サブスクモデルの初月イメージ

- 3,000 DL × 1.5% × ¥650 = `¥29,250` 粗売上
- 手数料後 = 約 `¥24,862`

- 10,000 DL × 2.0% × ¥650 = `¥130,000` 粗売上
- 手数料後 = 約 `¥110,500`

- 30,000 DL × 3.0% × ¥650 = `¥585,000` 粗売上
- 手数料後 = 約 `¥497,250`

### 解釈

- 単発ニーズなので、初期は買い切りのほうが相性がよい
- サブスクは「継続理由」を作れないと弱い
- 売上の本体は課金率よりも `自然流入 × App Store CVR` に強く依存する

## 12. 最終提案

最も筋がよい MVP は次。

- 既存の Web ロジックを共通化
- iOS では `Step 1-4 + Swipe + 好み可視化 + 保存`
- 課金はまず `買い切りプレミアム`
- ログインは後回し
- App Review は `4.2`, `3.1.1`, `5.1.1`, `2.1` を意識して準備

この構成なら、現状資産を活かしながら「ただの診断サイト」から「アプリらしい体験」にしっかり進化できる。
